# Hematite API docs

Static documentation site for [`api.hematite.gg`](https://api.hematite.gg),
generated by [`@pumicejs/docs`](https://www.npmjs.com/package/@pumicejs/docs)
and deployed to GitHub Pages.

This repository contains **no documentation source** of its own. It is a thin
build pipeline:

```
api.hematite.gg/@client  ──fetch──▶  pumice-docs generate  ──▶  dist/  ──▶  GitHub Pages
```

## Rebuild on demand

1. Open the **Actions** tab.
2. Pick **Build & deploy API docs** in the left rail.
3. Click **Run workflow** (top right).
4. Optionally override the `api_url` input. Defaults to `https://api.hematite.gg`.
5. Click the green **Run workflow** button.

The workflow also runs on a weekly cron so the site never drifts more than a
week behind the live API.

## Local development

Requires [Bun](https://bun.sh) `>=1.1`.

```bash
bun install

# Build against the live API (default https://api.hematite.gg)
bun run build

# Or point at a local API
API_URL=http://localhost:3000 bun run build

# Serve the result
bun run preview
```

Output lives in `./dist/`.

## First-time GitHub Pages setup

In the repo settings:

1. **Settings → Pages → Build and deployment → Source**: select **GitHub Actions**.
2. Run the workflow once via the **Actions** tab. The first deploy creates the
   `github-pages` environment automatically.
3. Optional: point a custom domain (e.g. `docs.hematite.gg`) at Pages and add
   a `CNAME` file under `public/` — or set it once in **Settings → Pages**.

## How it works

The build script (`scripts/build.mjs`) does four things:

1. `rm -rf dist`
2. `bunx pumice-docs generate -u $API_URL -f html -o dist`
3. Rewrites the absolute `/assets/*` paths in `dist/index.html` to relative
   `./assets/*` so the SPA works under any sub-path (project pages,
   custom domains, both).
4. Writes `.nojekyll` so GitHub Pages serves the bundle verbatim.

Everything else — the SPA, search, theming, code samples — comes prebuilt from
the `@pumicejs/docs` package.
